# Week 2: Building a RAG System from Scratch to API

This repository documents a structured, week-long deep dive into building a complete Retrieval-Augmented Generation (RAG) system. The project progresses from foundational theories to a fully functional, API-driven AI service.

The final result is a FastAPI application that exposes an endpoint (`/rag/ask`). This endpoint can answer questions based on a private knowledge base, demonstrating a practical, end-to-end implementation of a modern AI application.

## Final Application: Getting Started with the RAG API

These instructions explain how to run the final, containerized application from Day 14.

### 1. Prerequisites

*   Python 3.10+
*   `git` installed on your machine

### 2. Setup

First, clone the repository and navigate into the project directory:

```bash
git clone <YOUR_REPOSITORY_URL>
cd ai-engineer-02
```

Next, create and activate a virtual environment. This keeps project dependencies isolated.

```bash
# Create a virtual environment
python -m venv .venv

# Activate it (on macOS/Linux)
source .venv/bin/activate

# Or on Windows
# .venv\Scripts\activate
```

Create a `.env` file in the root directory to store your API key:

```
GOOGLE_API_KEY="YOUR_GOOGLE_API_KEY_HERE"
```

### 3. Installation

Install all the required Python packages using pip and the `requirements.txt` file (you will need to create this file first):

```bash
pip install -r requirements.txt
```

### 4. Running the Development Server

To run the API server, use Uvicorn from the project's root directory. The server will automatically load the RAG chain and make it available.

```bash
uvicorn app.main:app --reload
```

The API will now be running at `http://127.0.0.1:8000`.

### 5. Using the API

Interactive API documentation is automatically generated by FastAPI. You can explore and test the RAG endpoint by visiting:

*   **Swagger UI:** [http://127.0.0.1:8000/docs](http://127.0.0.1:8000/docs)

Navigate to the "RAG Service" section and the `POST /rag/ask` endpoint to ask questions.

## The 7-Day Learning Journey

This repository is the result of a daily, structured learning process. The individual practice scripts for each day are included to show the step-by-step evolution of the project.

### **Day 8: Foundational Theory**
*   **Focus:** Understanding the "why" behind RAG.
*   **Concepts:** Learned the limitations of standard LLMs (knowledge cut-offs, lack of private context, hallucinations), and how RAG solves this via an "open-book exam" approach. Studied the core concepts of **Embeddings** (turning text into semantic vectors) and **Vector Stores** (specialized databases for searching these vectors).

### **Day 9: Practical Embeddings**
*   **Focus:** Proving the concept of semantic similarity.
*   **Practice Script:** `embedding_practice.py`
*   **Concepts:** Used the `sentence-transformers` library to convert sentences into vectors. Implemented **Cosine Similarity** with `scikit-learn` to quantitatively prove that sentences with similar meanings have vectors that are mathematically "closer" than sentences with different meanings.

### **Day 10: Building the Knowledge Base**
*   **Focus:** Creating and querying a real vector store.
*   **Practice Script:** `chroma_practice.py`
*   **Concepts:** Used **ChromaDB** as a local, persistent vector store. Learned to create a "collection", populate it with documents (indexing), and perform a semantic `query()` to retrieve the most relevant text chunks for a given question.

### **Day 11: Talking to the LLM**
*   **Focus:** Isolating and mastering the "Generation" part of RAG.
*   **Practice Script:** `llm_practice.py` (and its improved versions)
*   **Concepts:** Learned to use the modern **Google GenAI SDK (`google.genai`)**. Mastered setting up API keys securely using environment variables (`.env`). Practiced making API calls to a specific model (`gemini-1.5-flash-latest`) and handling the response. Understood key generation parameters like `temperature`, `top_p`, and `top_k`.

### **Day 12: RAG from Scratch**
*   **Focus:** Assembling all individual components into a functional RAG pipeline without a framework.
*   **Practice Script:** `rag_from_scratch.py`
*   **Concepts:** Manually implemented the full **Retrieve-Augment-Generate** cycle:
    1.  **Retrieve:** Queried ChromaDB to get relevant context.
    2.  **Augment:** Constructed a detailed prompt template, combining the retrieved context with the user's question.
    3.  **Generate:** Sent the augmented prompt to the Gemini API to get a factual, context-aware answer.

### **Day 13: The Power of Abstraction (LangChain)**
*   **Focus:** Appreciating frameworks by rebuilding the RAG pipeline with significantly less code.
*   **Practice Script:** `langchain_rag.py`
*   **Concepts:** Learned the core building blocks of **LangChain**: `Retriever`, `ChatPromptTemplate`, and `ChatModel`. Mastered the **LangChain Expression Language (LCEL)**, using the `|` (pipe) operator to create a declarative and elegant data processing chain. Faced and solved real-world dependency update issues.

### **Day 14: Going Live with an API**
*   **Focus:** Integrating the smart RAG chain into a professional, web-accessible service.
*   **Final Code:** The `app/` directory.
*   **Concepts:** Combined Week 1's FastAPI skills with Week 2's AI logic. Implemented a professional project structure, used Pydantic for request/response validation, and exposed the LangChain RAG chain through a `POST /rag/ask` endpoint.

## Final Project Structure

The final, deployable application resides in the `app/` directory, reflecting a clean, scalable architecture.

```
/ai-engineer-02
├── .env                  # Stores secret API keys (ignored by Git)
├── .gitignore            # Specifies files for Git to ignore
├── requirements.txt      # Lists all project dependencies
├── README.md             # This file
├── app/                  # Main application package
│   ├── __init__.py
│   ├── main.py           # FastAPI app entrypoint
│   ├── models/
│   │   ├── __init__.py
│   │   └── rag_models.py # Pydantic request/response models
│   └── routers/
│       ├── __init__.py
│       └── rag.py        # APIRouter for the RAG service logic
└── _practice_scripts/    # (Recommended) Folder for all daily practice scripts
    ├── embedding_practice.py
    ├── chroma_practice.py
    └── ...
```

**Note:** It is recommended to move all the individual practice scripts (`embedding_practice.py`, `langchain_rag.py`, etc.) into a separate folder like `_practice_scripts/` to keep the root directory clean.
